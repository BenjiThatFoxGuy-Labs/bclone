# {{< icon "fa fa-paw fa-fw" >}} e621

`e621` is a filesystem view over [e621](https://e621.net) (and
e621ng-compatible instances such as [e926.net](https://e926.net) and
[e6ai.net](https://e6ai.net)), backed by the site's public JSON API. It's
primarily a **browser**: e621 has no folder tree of its own, so this
backend exposes a small, fixed set of virtual directories instead of a
real one, plus limited, opt-in uploading.

## Configuration

```sh
rclone config
```

```text
Storage> e621
e621 username.
username> yourusername
e621 API key.
api_key> XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
```

Generate the API key from "My Account" -> "Manage API Access" on e621.
Everything else is an advanced option with a sensible default — see
below.

## Path scheme: virtual directories

```text
e621:favorites (alias: favs)   posts tagged fav:me
e621:recent                    the most recent posts
e621:me                        posts tagged user:me (uploaded by the configured account)
e621:search/<tags>             posts matching the arbitrary tag search <tags>
e621:upload                    the one writable directory — see Uploading below
```

Every object, wherever it's listed or fetched from, is named
`<md5>.<ext>` — e621's own content hash and file extension.

A bare `<md5>.<ext>` can also be fetched directly (e.g. `rclone copyto
e621:<md5>.<ext> ./out.jpg`) by searching `md5:<md5>`, even outside the
directories above and without listing it first. This only works for
direct one-shot commands (`copy`, `copyto`, `cat`, ...), not while running
under `mount`/`serve`: those only ever see the directories actually
listed above, so an arbitrary `stat()` made while browsing a mount can't
turn into an e621 API search for a path that was never listed.

`search/<tags>` works the same way and is similarly left out of the root
listing — there's no fixed set of queries to enumerate — but it's usable
as the root of a `mount`/`union`/`serve` pointed directly at
`e621:search/<tags>`, since in that case the search directory *is* the
configured root rather than something that needs to be listed to be
reached. `<tags>` is e621's own space-separated tag search syntax; a
literal space in the path is enough, no URL-encoding needed.

## Uploading

Uploading a file writes it as a new post via `POST /uploads.json`, but
only through the `upload` directory at the root
(`e621:upload/<file>`) — no other path is writable, even when uploading
is otherwise available. Tags, rating and source aren't sent; the target
instance is expected to accept the file without them (e.g. into a
pending-post queue for review).

Uploading is always refused against the official e621.net, e926.net and
e6ai.net instances, regardless of configuration. Against any other
`endpoint`, it's available by default but can be turned off with
`disable_upload`. Either way, when uploading is unavailable the `upload`
directory itself doesn't exist: it's left out of the root listing, and
nothing can be written to it.

Since uploaded posts aren't tracked locally, `upload` always lists empty,
and a just-uploaded file can't be read back through this backend until
its post is resolved afresh (e.g. once it clears a pending-post queue) via
the usual `md5:<md5>` lookup.

## Standard options

Here are the standard options specific to e621 (API-based booru browser; upload support for non-official e621ng instances).

### --e621-username

e621 username.

Properties:

- Config:      username
- Env Var:     RCLONE_E621_USERNAME
- Type:        string
- Required:    true

### --e621-api-key

e621 API key.

Generate one from "My Account" -> "Manage API Access" on e621.

Properties:

- Config:      api_key
- Env Var:     RCLONE_E621_API_KEY
- Type:        string
- Required:    true

## Advanced options

Here are the advanced options specific to e621 (API-based booru browser; upload support for non-official e621ng instances).

### --e621-endpoint

The API endpoint to use.

Override this to target a different e621ng-compatible instance (for
example e926.net, or a self-hosted e621ng deployment) instead of the
default https://e621.net. This is arbitrary input: the backend makes a
best effort to talk to whatever endpoint is given, but only
e621ng-compatible instances are supported.

Uploading only works against a non-official instance: it's always refused
for e621.net, e926.net and e6ai.net, regardless of this setting (see
disable_upload to also turn it off elsewhere).

Properties:

- Config:      endpoint
- Env Var:     RCLONE_E621_ENDPOINT
- Type:        string
- Default:     "https://e621.net"

### --e621-disable-upload

Disable uploading even against a non-official, otherwise upload-capable endpoint.

Uploading is always refused against the official e621.net, e926.net and
e6ai.net instances regardless of this setting; set this to additionally
disable it for any other endpoint. Either way, when uploading is
unavailable the "upload" directory itself doesn't exist: it's left out of
the root listing, and nothing can be written to it.

Properties:

- Config:      disable_upload
- Env Var:     RCLONE_E621_DISABLE_UPLOAD
- Type:        bool
- Default:     false

### --e621-list-max

Maximum number of posts to consider when listing the favorites, favs, recent or me directories.

e621 paginates its post listings in pages of up to 320; listing walks
pages until this many posts have been seen. Raise it to see further back
into a large list, at the cost of more API calls per listing.

Properties:

- Config:      list_max
- Env Var:     RCLONE_E621_LIST_MAX
- Type:        int
- Default:     1000

## Limitations

- No modtime support (`SetModTime` always fails) and no directory
  deletion (`Rmdir` always fails; e621's API has no post-deletion call
  available to this backend).
- `favorites`/`favs`/`recent`/`me`/`search/<tags>` are read-only listings;
  `upload` is the only writable path, and it can't be listed back.
- A post whose file isn't visible to the requesting account (e.g. a
  deleted post without permission to view it) is skipped when listing,
  and fails to open if fetched directly.
